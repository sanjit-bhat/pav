what are the benefits of wish_getNextEp?
- wish_getNextep is basically a logical translation of the checks
the getNextEp func runs, along with the determ input->output computation.
- the benefits all boil down to: it bundles the checks
and determinism reasoning together, which makes the proof easier,
especially if you use wish_getNextEp in a wrapper Server postcond.
- e.g.,
in wp_getNextEp:
in err=true case, need to say that one of checks failed.
in err=false case, need to say that all checks passed.
wish_getNextEp de-duplicates the checks across the err branches.
- e.g.,
when getNextEp rets err, in Blame proof,
need to argue that Server passes all the checks.
this is easier when Server postcond bundles checks into wish_getNextEp.
- [most interesting] e.g.,
when doing check on getNextEp output, in Blame proof,
need to argue that Server passes the check.
this is easier when apply wish_getNextEp_det lemma to
align Client computation with Server postcond.
this hides the chain of determ internal to getNextEp.

there's a deeper issue going on here...
- core: our current setup has static boolean trust params.
a bunch of predicates / resources are hidden under a trust param being good.
i'll call these resources as "guarded" by the trust param.
e.g., guarded strong Server postcond.
e.g., guarded Client's digs align with Server's digs.
- with this setup, it's only easy to use guarded-resources
after learning that the trust param = true.
- this makes the proof annoying.
e.g., the Pav code has a bunch of error checks on untrusted Server data.
for each check, in the BlameSpec proof, we get to assume Server param = true.
and now, we get access to the un-guarded Server postcond,
and apply determinism lemmas to align Client computation
with the existential data in the Server's postcond.
after alignment, we finally have enough information to justify the error.
we have to repeat this same "computation alignment" for every error check,
and it only grows as more computation occurs.
- to reduce the annoyance, wish preds bundle determinism reasoning together,
so you only have to reason about determinism at the function boundary.
- but there might be other, smarter solutions.
e.g., ways to easily manipulate guarded resources.
